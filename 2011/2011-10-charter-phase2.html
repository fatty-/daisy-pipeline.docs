Overview

DAISY Consortium members and content producers have to keep up with an ever increasing set of digital content formats, all the more with the new revision of the DAISY standards and the rapid growth of EPUB usage in the consumer market. The demand for economic and efficient automated production tools is still very high.

The DAISY Pipeline 2 project, launched in June 2010, introduced a fundamental redesign of the original Pipeline 1 solution, enabling a more robust development of new functionality. It is a long term investment to achieve an enterprise-grade production system for accessible content.

After a first phase focusing on laying the foundations of the framework redesign and developing a first set of processing modules (with initial support for the new EPUB3 and DAISY Z39.86-201x "ZedAI" standard revisions), the present charter describes the second phase of the project. The development effort will now focus on consolidation of the first phase achievements, advanced framework functionality and deployment options, along with a new range of processing modules. It will notably enable TTS-based production of accessible content and coordinate with the Braille in DAISY initiative.

The outcome will benefit all DAISY Consortium members involved in automated content production, as well as commercial companies wishing to take advantage of the open source and liberally licensed deliverables.

Project Development Cycle

Based on the Board's decision that new core projects will not be started before the financing is secured, the Board of Directors of the DAISY Consortium decided in Seoul (June 2009) to follow a standardized Project Development Cycle for major projects and which consists of the following steps (see Minutes of the Board Meeting in Seoul, Items 9, 10, 49a, and 49b):

  1. Outline of a Project submitted by the management and approval by the Board for further development of the Project;
  2. Call for Participation (the members are invited to participate in the development of the corresponding Project Charter);
  3. Together with the finalized draft Charter a Call for Support is sent out to the membership; it asks for human and/or financial commitments;
  4. Submission of the final charter to the Board for final approval; it includes the budget and the agreements between the DAISY Consortium and the organizations regarding their commitments

The Outline for the Pipeline 2 project was approved by the Board at its meeting in Seoul in February 2009 (see Minutes of the Board Meeting in Seoul, Items 20 and 49f). The charter for the first phase of the project was approved by the Board at its meeting in Zurich in May 2010 (see Minutes of the Board Meeting in Zurich, Item 23 and Decision 06).

The present charter was edited by Romain Deltour integrating comments from Markus Gylling, Bernhard Heinser and George Kersher.

A call for support for the second phase was sent in Spring 2011. The present charter has been revised in October 2011 with the goal of being formally approved by the board in early November 2011.

Project Scope and Project Phases

This DAISY Pipeline 2 project essentially consists of two main tasks:

  1. the development of a core framework and runtime engine
  2. the development of several processing modules for automated production of accessible content

Note: because one of the main objectives of the project is to create an open framework for the creation of modular processing functions, the charter cannot obviously list all the future contributions explicitly. This charter only mentions the items primarily developed by the DAISY Consortium staff or of particular strategical importance.

A. First Phase: Results

The first phase of the project, covered by the previous charter, started in June 2010 and ended in September 2011. It encompassed the fundamental objectives towards a first usable release, focusing on the minimal framework redesign required to achieve a first reference implementation of the DAISY standard revision. It also includes the preparatory work required to plan the second phase, such as the creation of dedicated working groups for the development of Braille or TTS-based production.

The main deliverable of the first phase is the first public release of the Pipeline 2 project, version 1.0, and includes:

  • The foundations of the redesigned framework
    • based on third-party open source XProc and XSLT engines
    • designed with a modular architecture
    • promoting the use of testing toolkits and a common build infrastructure
    • executable via a command line interface
    • accessible by software applications via a Java API or an interoperable RESTful API (web service)
  • A set of processing modules based on standard and platform neutral XML processing technologies (XProc, XSLT, XPath)
    • DAISY 3 XML (DTBook) to DAISY 4 XML (ZedAI book profile)
    • DAISY 3 XML (DTBook) utility for upgrading to DTBook 2005-3
    • DAISY 3 XML (DTBook) utility for merging two or more files
    • DAISY 2.02 to EPUB 3
    • DAISY 4 XML (ZedAI) to EPUB 3
    • DAISY 4 XML (ZedAI) to HTML (the XHTML5 flavor used in EPUB) - not exposed in the user interface

The first phase results also include preparatory work carried out by the two independent TTS and Braille working groups.

B. Second Phase: September 2011 to September 2013

The second phase hereby chartered focuses on the consolidation of the first phase results, on the finalization of the framework redesign and on the development of new deployment options and new processing modules.

The following sections describe the objectives that can be reasonably achieved given the secured resources. An informative set of optional objectives are listed in an appendix as hints of what could be further developed with additional resources.

B.1 Framework and Deployment Options

B.1.1 Framework Internals

The DAISY Pipeline 2 core framework is the common basis for all the processing functions. It must be both complete and extensible enough to facilitate the implementation of all the planned and future developments.

The objectives are:

  1. Finalization of the component model for processing modules: a component model for the processing modules has been implemented in the first phase, although most of the modules can be run with vanilla XProc and XSLT engines. The overall objective is to facilitate the reuse of common components in several processing modules, for an increased flexibility and lower development cost. The component model will be further refined and consolidated in the second phase.
  2. Extension of the service-oriented approach within the framework: using OSGi (the de facto standard for a dynamic and service-oriented module system for Java) in the framework underpinnings increases modularity and maintainability. The service-oriented approach already implemented in the first phase will be further developed, notably for providing extension hooks to the underlying XProc, XSLT and XPath engines.
  3. Improvement of the configurability and maintainability of the framework: the framework developed in the first phase contains ancillary features like message logging, module management, etc. It will be ensured that a Pipeline 2 installation is easy to configure and maintain and update, possibly via the development of an administrative layer or the reuse of off-the-shelf tooling.
  4. Improvement of testing tools: tests must be easy to develop and execute and if possible automatically run by the build system. A solution for testing XProc pipelines must be adopted. The ongoing development of open source XProc and XSLT testing frameworks (XSPec, UTF-X) will be closely watched.

B.1.2 Web Service API

The development of a true interoperability layer is of utmost importance to the Pipeline 2 project. A first version of a Web Service API has been developed in the first phase. It provides a standard, platform-neutral way to access the DAISY Pipeline 2 functionality, allowing its use in distributed applications or its integration in broader systems. The second phase will gradually improve and refine the web service API.

  1. Refinement of the web service API: depending on user feedback and integration requirements, the web service API will be improved and finalized into a more comprehensive API. The final API may provide advanced features like on-the-fly event notifications, manipulation of the administrative layer, etc.
  2. Development of a security layer: security techniques such as authentication mechanisms must be added to the API in order to allow trusted calls to a remotely deployed Pipeline 2 service.
  3. Preparation for cloud deployment: the Pipeline 2 web service must be technically deployable on at least one public cloud facility (e.g. Amazon EC2, VMWare CloudFoundry, etc).

B.1.3 Developer Documentation

A comprehensive documentation is absolutely necessary to promote the adoption of the Pipeline 2 project and lower the learning curve for new developers.

  1. Write a comprehensive framework documentation: including architectural overviews, description of extensibility hooks, detailed API documentation, etc.
  2. Write developer guides: including setup documentation, various how-to guides, tips, best practices, etc.
  3. Write a developer charter: it shall provide general guidelines on how to be a good citizen in the Pipeline 2 developers community.

B.1.4 User Interface

The 1.0 release of the project only featured a command-line-based user interface. The second phase will start to move the focus towards ease of use with the development of alternative user interface options.

  • Development of a lightweight web-based UI: it must at the minimum provide a straightforward way to execute unique Pipeline jobs, and get the results and message reports. It can be seen as the equivalent of the lightweight GUI but as a web-based application. The web interface will allow for reduced admin and deployment costs in a centralized implementation.
  • Development of a lightweight graphical user interface (GUI): the user interface must be simple and easy to use, consisting in a few configuration dialogs, progress status and result reports. Additionally, it should be easily embeddable in a third-party application ("Save as DAISY"-plugins, DAISY authoring tools, etc.).

B.2 Processing Modules

B.2.1 EPUB 3 and Other Text-Only Production

Fully endorsed by the DAISY Consortium, EPUB 3 is set to become the leading format for accessible digital content. The production of quality and accessible EPUB 3 documents has become a primary objective of the Pipeline 2 project. The second phase will focus on:

  • Consolidation of EPUB 3 production modules: the modules developed in the first phase will be consolidated and gradually improved, in order to provide a rock-solid and versatile EPUB 3 production system. The modules that will be gradually improved are:
    • EPUB 3 utilities
    • ZedAI to XHTML5
    • ZedAI to EPUB 3
    • DAISY 2.02 to EPUB 3
  • EPUB 3 production for TTS-enabled reading systems: modules that add SSML pronunciation hints and PLS lexicons to EPUB 3 publications will be developed based on the utility modules developed for TTS-based production.
  • Development of new EPUB 3 production modules:
    • DTBook to EPUB 3: conversion from an DAISY 3 (ANSI/NISO Z39.86) XML document (DTBook) into an EPUB 3 publication
    • DAISY 3 to EPUB 3: conversion from a DAISY 3 (ANSI/NISO Z39.86) text-only or full-text/full-audio DTB into an EPUB 3 publication (with audio optionally converted as Media Overlays).
    • EPUB 2 to EPUB 3: conversion of EPUB 2 publications into EPUB 3 publications. Depending on the quality of the source EPUB 2, the conversion may not always be successful.
    • HTML 5 to EPUB 3: conversion of well-structured HTML documents into EPUB 3 publications. Depending on the quality of the source HTML document, the conversion may not always be successful.
    • EPUB 3 utilities: development of a set of utilities not provided in the first phase (e.g. generation of landmarks navigation, extraction/insertion of metadata, etc).
  • Development of other text-only production modules:
    • DTBook to HTML5: conversion of a DTBook document into a (set of) HTML 5 documents.

B.2.2 Validation Tools

Validation is critical to assess the quality of produced content. Third-party validation tools and XML grammar schemas can be integrated as Pipeline 2 modules to offer a unified access to the validation functionality.

The following modules will be provided:

  • DTBook validation: development of a schema-based DTBook validation module.
  • EPUB 3 validation: integration of epubcheck3 (under development at the time of writing) as a Pipeline 2 module
  • ZedAI validation: development of a schema-based ZedAI validation module.

B.2.3 TTS-based Production

One of the stated objective of the Pipeline 2 project is to develop a redesigned facility to better meet the popular need for TTS-based production of accessible content. A set of new modules will be developed following the preparatory work achieved by the dedicated TTS WG in the first phase of the project.

The common utility modules include:

  • a set of XML pre-processors: covering tasks like enhancement of the input XML markup, text normalization, etc.
  • a set of TTS adapters: enabling the use of a wide range of commercial and open source TTS engines, via several communication channels (SAPI5, web services, command line tools).
  • a set of components for audio processing: covering tasks like audio file merging, transcoding, etc.

Building on the above modules, the production functionality will include:

  • ZedAI to EPUB3 + Media Overlays: conversion of a DAISY 4 XML document (Z39.86-201X "ZedAI", at least the book profile) to an EPUB3 publication with audio content as Media Overlays.
  • DTBook to EPUB3 + Media Overlays: conversion of a DAISY 3 XML document (DTBook) to an EPUB3 publication with audio content as Media Overlays.

Note: the utility modules will also be used to enhance EPUB 3 content (SSML annotations, PLS lexicons) for use with TTS-enable reading systems, as specified in the B.2.1 section.

B.2.4 Braille Production

A working group has been created in the first phase to work towards obtaining Braille as a result of the production workflow. The collaborative DAISY Pipeline project offers a great opportunity for this group to integrate the driving forces and lead a pragmatic approach to Braille production.

Given the heterogeneity of technical requirements (notably due to locale specificities) and existing production tools, it would not make sense to focus on a single technical solution. Instead, the Braille Working Group will keep on supporting different approaches while maintaining a good level of synchronization and collaboration.

The Braille WG will initially work on a set of end-to-end prototype Braille production workflows integrated in the Pipeline 2. These prototypes should be supported by some Braille WG members on a volunteering basis. In this context, the objectives of the core Pipeline 2 developers team will be:

  • Development of pre-processing modules: covering tasks like enhancement of the input XML markup, text normalization, etc. Some of these modules will be the same as the modules developed in the context of TTS-based production.
  • Assist the Braille WG members with the integration of third-party Braille production tools: whenever possible, the team will assist with the development of the XProc or any other glue code that could be required to develop Pipeline 2 modules.

Team Organization and Management

The proposed project lead is Romain Deltour (DAISY Developer), reporting to Markus Gylling (DAISY CTO).

The project lead will supervise the core framework development and take special care of ensuring that the development of each separate processing function is harmonized with the overall architecture choices.

The development of processing functions will be handled by sub-groups with dedicated and optimized skill sets. Each group will have a group leader coordinating the work with the main project lead. The efficient management is based on close cooperation between the group leaders and the main project lead.

Conference calls will be scheduled bi-weekly for the core developer group, and upon request for sub-groups members and external contributors.

Strategic/Political Issues To Be Considered in the Project

Licensing

All the results of this project should be available royalty free and under a liberal licensing scheme that will encourage reuse of software components in other products (either commercial or open source).

As usual with open source developments, it must be noted that licensing decisions can have a strong impact on the the project. The reuse of existing open source libraries, to reduce the overall development cost, can only be viable if the licenses of these dependencies are compatible with the license chosen for the final deliverable. In an extreme scenario, the impossibility to use a library due to licensing incompatibilities can lead to major additional development costs which may jeopardize the budget and timeline defined in the present charter.

To minimize the risk of legal dead-end, the licensing decisions will be taken early in the project development phase and will require the input of both developers and legal professionals.

Role of the DAISY Consortium

It is strategically important to underline the widely collaborative aspect of the DAISY Pipeline project. The DAISY Consortium is the umbrella organization that maintains and promotes the project, but pieces of functionality are expected to be contributed by third-party organizations. In this context, it must be noted that while the DAISY Consortium will support the core framework it cannot systematically allocate resources to the maintenance and further development of organization-specific contributions. These tasks are rather considered part of the commitments of said organization.

Relationship with the Pipeline 1

Because the Pipeline 2 includes a fundamental redesign of the Pipeline 1 framework, it is worth mentioning that both solutions will co-exist for a significant amount of time. External contributions based on the new framework are expected to progressively increase and eventually outnumber the contributions to the Pipeline 1, but until then this duality puts high demand on the DAISY Consortium staff for technical maintenance and support.

"Enabling Technologies Framework" Project

The Enabling Technologies Framework is a three year project funded by the World Intellectual Property Organization (WIPO) and endorsed by the Stakeholders Platform of WIPO. It is a joint project involving both EDItEUR and the DAISY Consortium. Its general objective is to evolve mainstream publishing processes to enable the production of fully accessible digital content.

The DAISY Pipeline 2 plays a major role as one of the work packages is entirely dedicated at extending the DAISY pipeline to support the requirements of publishers and conversion houses. This notably involves the participation of the publishing community in a requirements gathering phase.

The project's second year deliverables covered by this charter include:

  • Project plans, developed in collaboration with WIPO technical staff, for the implementation of the DAISY Pipeline 2 software within WIPO.
  • DAISY Pipeline 2 software and its tool sets mounted and operational on the WIPO web site. To be scheduled in concert with WIPO (probably for the 1st quarter 2012).
  • Statistical information on the downloads or usage of Pipeline 2 software from the DAISY website.
  • Additional services:
    • Three days on-site assistance to the technical staff of WIPO.
    • Creation of a web-based forum for Pipeline 2 software questions and answers.
    • EMail technical support to WIPO
    • Limited telephone or Skype support to WIPO

Risk Management

Threat Probability Strategy Action
The license of a desired dependency is incompatible with the project license Medium Acceptance Either decide to change the project license or integrate some additional development effort in the budget and timeline
Resources are insufficient to achieve the project objectives Medium Mitigation As the framework stabilizes we have a better measure of what the development costs can be. The level of possible further development will depend on additional funding.
Contributor withdrawal leads to discontinuation of certain functionality Medium to High Acceptance Given the nature of the project as a framework/host for multiple transformation types, it is a natural risk that certain functionality is discontinued when and if there is no continued community support for it. Under normal circumstances, DC staff does not take on the task of maintaining functionality originally committed by other contributors. This general rule does not apply to functionality of extra high long term strategic importance, for which DC staff may step in to assure longevity and stability.
Uptake in commercial sector does not meet expectations Low Mitigation Uptake in commercial sector can be stimulated through dedicated outreach efforts.

Appendix: Optional Objectives

This section is informative: the objectives hereafter described cannot be reasonably achieved without further resource contributions ; they are not part of the expected deliverables of the second phase of the Pipeline 2 project. Most of them can be seen as buckets for extended developments, which can be refined and chartered as satellite sub-projects depending on the interest of the additional contributors.

Framework and APIs

  1. Development tools: tools can be developed to help developers with both tedious one-time tasks and day-to-day chores, including test drivers (possibly for the IDE), scripts for quickly setting up skeleton code for the development of a new module (rapid application development), etc.
  2. Sample code using the Pipeline web service API: in order to lower the cost of integrating the Pipeline in a broader system, a few applications can be developed and demo how to call the Pipeline from different contexts (e.g. a .Net application calling the Web Service API, a Java application calling the Java API, etc.).

User Interface

  • Graphical user interfaces for the admin and configuration layer: as a separate application or integrated into another GUI, it would provide the user interfaces for handling all the admin tasks (e.g. installing a new processing module) and internal configuration options (e.g. customizing the TTS voices configuration, editing an acronym lexicon).
  • Rich graphical user interface (GUI): the interface could allow the manipulation of either a local installation (via the Java API) or centralized installation (via the WS API) of the Pipeline 2. It could be developed on the basis of the existing Pipeline GUI project or be totally redesigned.
  • Web-based UI for the admin and configuration layer: it would provide the user interfaces for handling all the admin tasks (e.g. installing a new processing module) and internal configuration options (e.g. customizing the TTS voices configuration, editing an acronym lexicon).
  • Fully featured web application: it would enable more complex use cases, including the management of user accounts, a storage area, use of email services, etc. It could be developed on the basis of the existing PipeOnline project or as a new application.

Text-Only Production

  • Advanced EPUB utilities: e.g. resource/font obfuscation, XML digital signature, etc.
  • ZedAI to RTF: conversion of a DAISY 4 XML document (Z39.86-201X "ZedAI") to Microsoft Rich Text Format.
  • Large-pring production: conversion of a DAISY 4 XML document (Z39.86-201X "ZedAI") to large-pring (e.g. via LaTeX, or via HTML+CSS).
  • DocBook from/to ZedAI: bi-directional conversion of a !DocBook5 document to a DAISY 4 XML document (Z39.86-201X "ZedAI").
  • ZedAI to DTBook: conversion of a DAISY 4 XML document (Z39.86-201X "ZedAI") to a DAISY 3 XML (DTBook) document.

Validation

  • DAISY 3 validation: integration of ZedVal as a Pipeline 2 module, after a cleanup of ZedVal library dependencies
  • DAISY 2.02 validation: port of the Pipeline 1 DAISY 2.02 light validator

TTS-based Production

  • EPUB3 to EPUB3/Audio: conversion of a text-only EPUB3 publication to an EPUB3 publication with audio content as Media Overlays.
  • ZedAI to DAISY 3: conversion of a DAISY 4 XML document (Z39.86-201X "ZedAI", at least the book profile) to a DAISY 3 (Z39.86-2005) full-text full-audio DTB.
  • ZedAI to DAISY 2.02: conversion of a DAISY 4 XML document (Z39.86-201X "ZedAI", at least the book profile) to a DAISY 2.02 full-text full-audio DTB.
